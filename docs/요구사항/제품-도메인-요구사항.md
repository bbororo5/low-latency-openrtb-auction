# 제품·도메인 요구사항

상태: 개편 초안

수치와 합격 기준: [부하·데이터·검증 기준](부하-데이터-검증-기준.md)

이 프로젝트는 제한 시간 경매, 자사 DSP의 캠페인 선택, 광고주 예산 정합성, 장애 지속성을 구현·검증한다.

## 1. 범위

| 역할 | 책임 |
|---|---|
| 공급자 | 광고 기회와 제한 시간을 SSP에 전달하고 결과를 받는다. 클라이언트 노출을 확인하면 과금 가능 노출을 비동기로 통지한다. |
| SSP | 요청 검증, DSP 호출, 입찰 검증, 낙찰자 결정을 담당한다. |
| 자사 DSP | 캠페인·소재·예산으로 입찰 여부와 가격을 결정한다. |
| 외부 DSP | 서로 다른 회사를 나타내며 독립적으로 응답한다. |
| 광고주 | 자사 DSP에 캠페인과 예산을 맡긴다. |

범위에는 하나의 광고 영역, 자사 DSP와 두 외부 DSP, 실제 캠페인 데이터, 잠재 지출·낙찰·패찰·과금·만료, 인스턴스·AZ 장애가 포함된다.

다음은 범위 밖이다.

- 광고 표시 SDK, 클릭·기여 추적, 사용자 식별·프로파일링
- 캠페인·소재 원본 관리 기능과 관리 화면
- 머신러닝 입찰 최적화, 고급 속도 조절, 빈도 제한
- 광고 사기·가시성·소재 심사·브랜드 안전성
- 복수 통화, 청구·환불, 대외 정산·회계
- 외부 사업자 온보딩·공개망 보안 운영과 외부 DSP 내부
- 다중 리전 재해 복구, 쿠버네티스, 자동 확장 제어면

## 2. 핵심 요구사항

### `요구-01` 제한 시간 경매

SSP는 요청을 검증하고 자사 DSP와 두 외부 DSP에 의미상 같은 OpenRTB 2.6 요청을 보낸다. 각 DSP 호출을 `VALID_BID`, `NO_BID`, `TIMEOUT`, `INVALID_BID`, `ERROR` 중 하나로 끝내고 유효 입찰만으로 낙찰자 또는 낙찰 없음을 결정한다.

- `입찰 마감 = SSP 요청 수신 시각 + 유효 tmax`이다.
- DSP에는 입찰 마감까지 남은 시간만 전달하며 마감 이후 응답은 결과를 바꾸지 않는다.
- 입찰은 요청, 광고 영역, 가격, 통화, 매체, 소재 기준을 모두 통과해야 한다.
- 유효 입찰 중 CPM이 가장 높은 입찰이 낙찰하며 동률은 `dspId`, `bidId` 순으로 결정한다.

검증: `검증-01`, `검증-02`, `검증-03`

### `요구-02` 캠페인 입찰

자사 DSP는 실행용 데이터에서 요청 조건과 예산을 만족하는 캠페인·소재를 선택하고 결정적인 가격으로 입찰한다.

1. 운영 기간, 매체·광고 형식, 크기, 게재 위치·분류, 최저가, 통화로 후보를 거른다.
2. 광고주와 캠페인의 사용 가능 예산을 확인한다.
3. `금일 과금 노출 수 / 금일 목표 노출 수`가 가장 낮은 캠페인을 선택한다.
4. 동률은 `campaignId`, `creativeId` 순으로 결정한다.
5. 적격 후보가 없으면 `NO_BID`한다.

검증: `검증-01`, `검증-02`

### `요구-03` 예산과 과금

자사 DSP는 입찰을 반환하기 전에 광고주·캠페인 예산 범위의 잠재 지출을 확보한다. SSP는 공급자의 과금 통지를 해당 경매·입찰에 연결하고, 자사 DSP는 잠재 지출을 확정 지출로 전환한다. 낙찰·패찰·과금·만료는 중복과 순서 변경에 안전하게 처리한다.

- 통화는 KRW 하나고 입찰가와 최저가는 CPM이다. 노출 1건 금액은 CPM의 `1/1000`이다.
- 가격 비교와 예산 계산에서 정밀도 손실이 발생하지 않아야 한다. 허용 정밀도와 내부 표현 방식은 아키텍처 결정에서 정한다.
- 일일 예산의 날짜는 경매 수신 시각의 `Asia/Seoul` 날짜다.

```text
광고주 확정 지출 + 잠재 지출 <= 충전액
캠페인 누적 확정 지출 + 잠재 지출 <= 누적 한도
캠페인 일일 확정 지출 + 잠재 지출 <= 일일 한도
```

| 이벤트 | 금액 효과 |
|---|---|
| 입찰 제시 | 노출 1건 금액을 잠재 지출로 잡는다. |
| 낙찰 | 낙찰을 확정하지만 확정 지출은 늘리지 않는다. |
| 패찰·만료 | 잠재 지출을 해제한다. |
| 과금 노출 | 낙찰가 만큼 잠재 지출을 줄이고 같은 금액을 확정 지출로 한 번만 전환한다. |

사용 가능 예산은 `한도 - 확정 지출 - 잠재 지출`이다. 과금은 잠재 지출을 확정 지출로 재분류하므로 사용 가능 예산을 다시 차감하지 않는다.

동일한 `auctionId + bidId + eventType`은 금액에 한 번만 영향을 준다. 낙찰과 과금의 순서가 바뀌어도 최종 상태는 같아야 하며 패찰과 과금처럼 모순된 종료 이벤트는 `MONEY_EVENT_CONFLICT`로 격리한다.

검증: `검증-01`, `검증-04`

### `요구-04` 부분 실패와 과부하

하나의 DSP 지연·오류는 다른 DSP 호출과 경매 전체를 중단시키지 않는다. 스레드, 대기열, 연결, 처리 중 작업에 상한을 두고 과부하 후 재시작 없이 회복한다. 부하가 증가해도 늦거나 잘못된 입찰이 낙찰하거나 예산 불변식이 깨지면 안 된다.

검증: `검증-03`, `검증-05`

### `요구-05` 장애 지속성과 상태 보존

SSP 인스턴스 하나, 자사 DSP 인스턴스 하나, AZ 하나의 중단이 전체 서비스 중단을 만들면 안 된다. 순차 배포는 전체 중단과 금액 불일치 없이 끝나야 한다.

- 장애 전에 인정한 금액 효과는 유실·중복되면 안 되며 RPO는 0이다.
- 장애 순간의 처리 중 경매는 일부 실패할 수 있지만 예산 불변식은 깨지면 안 된다.
- 리전 전체 장애는 범위 밖이다.

검증: `검증-06`, `검증-07`, `검증-08`

### `요구-06` 검증 가능성

경매, DSP, 캠페인 검색, 금액, 자원 포화, 장애 전환을 관측하고 성능·장애·예산 대조 결과를 재현할 수 있어야 한다.

검증: `검증-01`~`검증-08`

### `요구-07` 통신 형식 독립성

JSON을 이진 표현으로 바꾸어도 경매·금액 의미와 합격 시험은 달라지지 않는다.

검증: `검증-01`

## 3. 제약과 가정

제약:

- SSP–DSP 간 의미는 OpenRTB 2.6의 명시적 하위 규격을 따른다.
- 공급자 API는 프로젝트 전용이며 OpenRTB 객체를 직접 노출하지 않는다.
- 첫 버전은 하나의 광고 영역, 1차 가격, 배너 우선, KRW 단일 통화다.
- 실행 환경은 Java 21이다.
- AWS 단일 리전에서 AZ 하나의 장애를 견뎌야 하며 SSP와 자사 DSP의 단일 애플리케이션 인스턴스 배포는 허용하지 않는다.
- HTTP/JSON은 초기 표현일 뿐 영구 제약이 아니다.

가정:

- 외부 DSP의 내부 캠페인과 예산은 알 수 없다.
- 매체·캠페인 원본은 별도 시스템이 관리하며 재현 가능한 실행용 데이터를 제공한다.
- 공급자는 클라이언트 노출 결과를 확인한 뒤 경매·입찰 식별자, 낙찰가, 통화를 포함한 신뢰할 수 있는 서버 측 과금 통지를 제공한다.
- 일일 예산의 날짜 경계는 `Asia/Seoul`이다.

구현이 어렵다는 이유로 기준을 낮추지 않는다. 예산 불변식이나 장애 범위를 바꾸려면 ADR이 아니라 이 문서부터 바꾼다.
