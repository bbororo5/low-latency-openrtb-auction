# 제품·도메인 요구사항

상태: 개편 초안

수치와 합격 기준: [부하·데이터·검증 기준](workload-data-verification.md)

이 프로젝트는 제한 시간 경매, 자사 DSP의 캠페인 선택, 광고주 예산 정합성, 장애 지속성을 구현·검증한다.

## 1. 범위

| 역할 | 책임 |
|---|---|
| 공급자 | 광고 기회를 요청하고 렌더링 결과를 SSP에 통지한다. 별도 애플리케이션 없이 k6와 통합 시험이 역할을 모의한다. |
| SSP | 요청 검증, DSP 호출, 낙찰 입찰 결정, 렌더링 검증, 과금·미과금 확정을 담당한다. |
| 자사 DSP | 캠페인·소재·예산으로 입찰을 결정하고 SSP의 최종 결과에 따라 잠재 지출을 확정하거나 해제한다. |
| 외부 DSP | 서로 다른 회사를 나타내며 각자 독립된 모의 시스템으로 응답한다. |
| 광고주 | 자사 DSP에 캠페인과 예산을 맡긴다. |

공급자, SSP, 자사 DSP, 두 외부 DSP는 서로 다른 회사다. SSP와 자사 DSP는 독립 배포·독립 저장소를 사용하고 계약으로만 통신한다. SSP와 자사 DSP의 핵심 거래 경로는 제품 수준으로 구현하고 공급자와 외부 DSP는 시험에 필요한 범위만 모의한다.

범위에는 하나의 광고 영역, 자사 DSP와 두 외부 DSP, 실제 캠페인 데이터, 잠재 지출·낙찰·패찰·과금·미과금, 인스턴스·AZ 장애가 포함된다.

다음은 범위 밖이다.

- 공급자 애플리케이션, 광고 표시 SDK, 실제 브라우저 렌더링
- 클릭·기여 추적, 사용자 식별·프로파일링
- 캠페인·소재 원본 관리 기능과 관리 화면
- 머신러닝 입찰 최적화, 고급 속도 조절, 빈도 제한
- 광고 사기·가시성·소재 심사·브랜드 안전성
- 복수 통화, 청구·환불, 대외 정산·회계
- 외부 사업자 온보딩·공개망 보안 운영과 외부 DSP 내부
- 다중 리전 재해 복구, 쿠버네티스, 자동 확장 제어면

## 2. 핵심 요구사항

### `요구-01` 제한 시간 경매

SSP는 요청을 검증하고 자사 DSP와 두 외부 DSP에 의미상 같은 OpenRTB 2.6 요청을 보낸다. 각 DSP 호출을 `VALID_BID`, `NO_BID`, `TIMEOUT`, `INVALID_BID`, `ERROR` 중 하나로 끝내고 유효 입찰만으로 낙찰자 또는 낙찰 없음을 결정한다.

- `입찰 마감 = SSP 요청 수신 시각 + 유효 tmax`이다.
- DSP에는 입찰 마감까지 남은 시간만 전달하며 마감 이후 응답은 결과를 바꾸지 않는다.
- 입찰은 요청, 광고 영역, 가격, 통화, 매체, 소재 기준을 모두 통과해야 한다.
- 유효 입찰 중 CPM이 가장 높은 입찰이 낙찰하며 동률은 `dspId`, `bidId` 순으로 결정한다.
- DSP 하나가 지연·실패해도 다른 유효 입찰로 기한 안에 경매를 끝내며 유효 입찰이 없으면 광고 없음으로 응답한다.
- 동일한 `auctionId + impressionId`는 하나의 논리적 경매이며 최대 한 건의 낙찰 입찰과 한 건의 과금 결과만 만든다.
- 중복 공급자 요청은 추가 캠페인 선택·잠재 지출·낙찰·과금을 만들지 않는다.

검증: `검증-01`, `검증-02`, `검증-03`

### `요구-02` 캠페인 입찰

자사 DSP는 실행용 데이터에서 요청 조건과 예산을 만족하는 캠페인·소재를 선택하고 결정적인 가격으로 입찰한다.

1. 운영 기간, 매체·광고 형식, 크기, 게재 위치·분류, 최저가, 통화로 후보를 거른다.
2. 광고주와 캠페인의 사용 가능 예산을 확인한다.
3. `금일 과금 노출 수 / 금일 목표 노출 수`가 가장 낮은 캠페인을 선택한다.
4. 동률은 `campaignId`, `creativeId` 순으로 결정한다.
5. 적격 후보가 없으면 `NO_BID`한다.

동일한 `auctionId + impressionId`가 다시 들어오면 최초 캠페인·소재·가격·`bidId`와 잠재 지출을 재사용한다. 최초 결과를 안전하게 확인할 수 없으면 새 입찰을 만들지 않고 `NO_BID`한다.

검증: `검증-01`, `검증-02`

### `요구-03` 예산과 과금

자사 DSP는 입찰을 반환하기 전에 광고주·캠페인 예산 범위의 잠재 지출을 확보한다. SSP는 렌더링 통지를 경매·낙찰 입찰에 연결해 과금 또는 미과금을 최종 확정한다. 자사 DSP는 SSP의 권위 있는 최종 결과를 확인한 뒤에만 잠재 지출을 확정하거나 해제한다.

- 통화는 KRW 하나고 입찰가와 최저가는 CPM이다. 노출 1건 금액은 CPM의 `1/1000`이다.
- 가격 비교와 예산 계산에서 정밀도 손실이 발생하지 않아야 한다. 허용 정밀도와 내부 표현 방식은 아키텍처 결정에서 정한다.
- 일일 예산의 날짜는 경매 수신 시각의 `Asia/Seoul` 날짜다.

```text
광고주 확정 지출 + 잠재 지출 <= 충전액
캠페인 누적 확정 지출 + 잠재 지출 <= 누적 한도
캠페인 일일 확정 지출 + 잠재 지출 <= 일일 한도
```

| 이벤트 | 금액 효과 |
|---|---|
| 입찰 제시 | 노출 1건 금액을 잠재 지출로 잡는다. |
| 낙찰 통지 | 잠재 지출을 유지한다. |
| 패찰·SSP 미접수·미과금 확정 | 잠재 지출을 해제한다. |
| 과금 확정 | 낙찰가만큼 잠재 지출을 줄이고 같은 금액을 확정 지출로 한 번만 전환한다. |

사용 가능 예산은 `한도 - 확정 지출 - 잠재 지출`이다. 과금은 잠재 지출을 확정 지출로 재분류하므로 사용 가능 예산을 다시 차감하지 않는다.

동일한 `auctionId + impressionId + bidId + eventType`은 금액에 한 번만 영향을 준다. 모든 잠재 지출은 SSP 미접수, 패찰, 과금 또는 미과금이라는 권위 있는 결과로 종료되어야 한다. DSP는 자체 시간 만료만으로 잠재 지출을 해제하지 않는다.

SSP는 하나의 입찰에 모순된 최종 결과를 만들지 않는다. DSP가 모순된 결과를 받으면 금액을 자동으로 다시 변경하지 않고 `MONEY_EVENT_CONFLICT`로 격리한다.

검증: `검증-01`, `검증-04`

### `요구-04` 부분 실패와 과부하

하나의 DSP 지연·오류는 다른 DSP 호출과 경매 전체를 중단시키지 않는다. 스레드, 대기열, 연결, 처리 중 작업에 상한을 두고 과부하 후 재시작 없이 회복한다. 부하가 증가해도 늦거나 잘못된 입찰이 낙찰하거나 예산 불변식이 깨지면 안 된다.

검증: `검증-03`, `검증-05`

### `요구-05` 통지와 최종 정산

각 DSP는 입찰 응답에 OpenRTB의 `nurl`, `lurl`, `burl`을 제공하고 SSP는 사전 등록된 주소만 경매 경로 밖에서 비동기로 호출한다.

- `nurl`은 낙찰 사실만 알리며 금액을 바꾸지 않는다.
- `lurl`은 패찰한 입찰의 잠재 지출을 해제한다.
- `burl`은 과금 확정 결과를 전달한다.
- 낙찰 후 렌더링되지 않은 입찰의 미과금 결과도 DSP가 최종 확인할 수 있어야 한다. 전달 방식은 아키텍처 결정에서 정한다.
- DSP의 성공 응답은 통지를 장애 후에도 복구 가능한 형태로 접수했음을 뜻한다.
- 정상 상황에서 SSP의 최종 확정부터 DSP의 복구 가능한 접수까지 p99는 5초 이하다.

과금 가능 시간은 SSP의 낙찰 확정부터 60초다. SSP가 복구 가능하게 접수한 최초 시각이 기한 이하면 과금하고 이후면 미과금한다. 클라이언트가 주장한 시각과 처리 실행 순서는 판정을 바꾸지 않는다.

렌더링 통지는 SSP가 발급한 경매 결과, 낙찰 입찰과 연결되고 위조·중복이 방지된 경우만 인정한다. 성공 응답은 사건 식별자와 최초 접수 시각이 장애 후에도 복구 가능함을 뜻한다.

검증: `검증-04`, `검증-09`

### `요구-06` 장애 지속성과 상태 보존

SSP 인스턴스 하나, 자사 DSP 인스턴스 하나, AZ 하나의 중단이 전체 서비스 중단을 만들면 안 된다. 순차 배포는 전체 중단과 금액 불일치 없이 끝나야 한다.

- 장애 전에 인정한 금액 효과는 유실·중복되면 안 되며 RPO는 0이다.
- 장애 순간의 처리 중 경매는 일부 실패할 수 있지만 예산 불변식은 깨지면 안 된다.
- 외부에 공개한 경매 결과, 성공으로 접수한 렌더링 통지, 최종 정산 결과, 미전달 통지와 멱등성 근거는 인스턴스·AZ 장애 뒤에도 복구해야 한다.
- 실행 중이던 경매 계산은 복구하지 않지만 이미 만들어진 자사 DSP 잠재 지출은 권위 있는 결과로 수습해야 한다.
- 메모리 타이머가 사라져도 과금 기한이 지난 낙찰을 찾아 미과금으로 확정할 수 있어야 한다.
- 리전 전체 장애는 범위 밖이다.

검증: `검증-06`, `검증-07`, `검증-08`

### `요구-07` 검증 가능성

경매, DSP, 캠페인 검색, 금액, 자원 포화, 장애 전환을 관측하고 성능·장애·예산 대조 결과를 재현할 수 있어야 한다.

검증: `검증-01`~`검증-09`

## 3. 제약과 가정

제약:

- SSP–DSP 간 의미는 OpenRTB 2.6의 명시적 하위 규격을 따른다.
- 공급자 API는 프로젝트 전용이며 OpenRTB 객체를 직접 노출하지 않는다.
- 첫 버전은 하나의 광고 영역, 1차 가격, 배너 우선, KRW 단일 통화다.
- 실행 환경은 Java 21이다.
- AWS 단일 리전에서 AZ 하나의 장애를 견뎌야 하며 SSP와 자사 DSP의 단일 애플리케이션 인스턴스 배포는 허용하지 않는다.
- HTTP/JSON은 초기 표현일 뿐 영구 제약이 아니다.
- SSP와 자사 DSP는 상대 저장소를 공유하거나 내부 상태를 직접 읽고 변경하지 않는다.

가정:

- 외부 DSP의 내부 캠페인과 예산은 알 수 없다.
- 매체·캠페인 원본은 별도 시스템이 관리하며 재현 가능한 실행용 데이터를 제공한다.
- 공급자 역할은 k6와 통합 시험이 수행하며 공급자 요청·렌더링 통지 API는 SSP에 구현한다.
- 렌더링 통지는 과금 사실을 주장할 뿐 확정 권한은 SSP에 있다.
- 일일 예산의 날짜 경계는 `Asia/Seoul`이다.

구현이 어렵다는 이유로 기준을 낮추지 않는다. 예산 불변식이나 장애 범위를 바꾸려면 ADR이 아니라 이 문서부터 바꾼다.
