# 부하·데이터·검증 기준

상태: 확정안

상위 문서: [제품·도메인 요구사항](product.md)

이 문서는 부하·데이터·장애 구간과 합격 기준을 정한다. 복제본 수, 저장소와 제어 알고리즘은 정하지 않는다.

## 1. 트래픽 구간

일 1천만은 공급자 경매 요청 수이며 특정 회사 실측이 아니라 포트폴리오 사업 가정이다. 요청 하나에 여러 슬롯이 있으면 실제 슬롯 경매 수는 더 커지므로 시험 결과에 슬롯 수를 별도로 기록한다.

```text
일평균 = 10,000,000 / 86,400 = 115.74 RPS
바쁜 시간 평균 = 5,000,000 / 14,400 = 347.22 RPS
설계 정점 = 400 RPS
정상 첨두 검증 = 500 RPS
순간 과부하 = 1,000 RPS
```

| 구간 | 요청률 | 시간 | 의미 |
|---|---:|---:|---|
| 일평균 | 115.74 RPS | 24시간 환산 | 사업 물량 산정 |
| 설계 정점 | 400 RPS | 바쁜 4시간 | 장애·과부하에서도 보호할 처리량 |
| 정상 첨두 | 500 RPS | 10분 | 정상 상태 성능 합격 |
| 순간 과부하 | 1,000 RPS | 60초 | 빠른 거절과 회복 검증 |

SSP는 공급자 요청마다 프로젝트 DSP 하나와 외부 DSP 두 회사에 병렬로 요청한다. 슬롯 수는 아직 고정 평균으로 가정하지 않고 시험마다 명시한다.

## 2. 시간 목표

- 공급자 관측 경매 시간의 p99는 정상 첨두와 외부 DSP 하나의 실패 상황에서 50ms 이하다.
- `tmax=180ms`는 공급자의 절대 상한이며 50ms 성능 목표와 다르다.
- 180ms 뒤의 응답과 DSP 입찰은 무효다.
- 1,000 RPS에서는 최소 400 RPS를 p99 50ms 안에 보호하고 초과 요청은 빠른 기술 실패로 거절한다. 전체 도착 요청의 p99 목표는 두지 않는다.
- 부하가 100 RPS로 감소한 뒤 30초 안에 정상 범위로 회복한다.
- 400 RPS에서 인스턴스 장애는 10초, AZ 장애는 30초 안에 성공률 99.9%와 p99 50ms를 회복한다.
- 리전 하나가 중단되면 다른 리전의 SSP 경매와 외부 DSP 호출은 계속되고, 프로젝트 DSP는 이미 안전하게 할당된 예산 범위에서 입찰을 계속한다. 전체 중단, 예산 초과와 중복 과금은 허용하지 않는다.

50ms는 k6 호출자에서 본 서버 간 경매 시간이다. 부하 발생기에서 SSP 진입까지의 네트워크, 부하 분산, SSP, 병렬 DSP 호출, 프로젝트 DSP 게이트웨이, 캠페인 선택·예약과 응답 전송을 포함한다. DNS와 최초 연결 수립은 제외하고 준비된 재사용 연결을 사용한다. 비동기 통지·렌더링·과금은 제외한다.

## 3. 데이터 기준

| 데이터 | 규모·분포 |
|---|---|
| 공급자 | 1,000개 |
| 게재 위치 | 10,000개, 상위 20%가 요청 80% 처리 |
| 광고주 | 10,000개, KRW |
| 캠페인 | 100,000개, 활성 30,000개 |
| 소재 | 200,000개, 캠페인당 평균 2개 |
| 활성 후보 | 요청당 p50 20 / p95 200 / p99 1,000 |
| 예산 편향 | 상위 1% 캠페인에 금액 연산 20% 집중 |
| 소진 임박 | 활성 캠페인 1%는 노출 1~10건 금액만 남음 |

시험 CPM은 정수와 소수 셋째 자리 경계를 모두 포함한다. 캠페인은 총예산, 고정 CPM, 시작·종료 시각과 소재를 가지며 광고주 충전액과 일일 예산은 만들지 않는다.

## 4. 시험 분리

### 용량 시험

500 RPS를 10분간 유지한다. 캠페인 예산은 시험 중 페이싱 때문에 처리량이 제한되지 않도록 충분히 크게 잡고, p99 50ms와 정확성을 판정한다.

### 페이싱 시험

통제된 요청과 가상 시간을 사용한다. 캠페인 총예산은 페이싱이 없으면 전체 기간의 25% 이내에 소진되도록 역산한다. 적격 기회가 충분한 조건에서 선형 목표를 따르고 종료까지 예산을 분배하는지 확인한다.

허용 편차는 다음 하한과 사업 목표를 함께 계산해 시험 전에 고정한다.

```text
최소 제어 편차 ≈ 양 리전의 캠페인별 최대 미결 리스 권한 합계
               + 최대 지출 속도 × 리스 발급·반환 제어 주기
```

리스 크기·유효 기간·보충 임계치와 허용 편차는 시험 전에 함께 고정한다. 리스가 작고 짧을수록 편차는 줄지만 리전 원장 통신과 경합은 증가한다.

### 통합 시험

용량과 페이싱을 각각 통과한 뒤 500 RPS에서 잠재 지출, 패찰, 과금, 만료와 페이싱을 함께 실행한다. 낙찰률·렌더링률은 업계 통계로 주장하지 않고 모든 상태 전이를 재현하기 위한 명시적 시험 입력으로 기록한다.

## 5. 장애·결함 입력

| 조건 | 기대 결과 |
|---|---|
| 중복 공급자·DSP·렌더링 통지 | 경매·예약·금액 효과 한 번 |
| 입찰 응답 유실 | 해당 잠재 지출이 예약 후 5초 안에 만료 |
| `lurl` 응답 유실 | 재호출되면 금액 해제 한 번, 재호출되지 않아도 5초 만료로 해제 |
| `burl` 응답 유실 | 재호출되면 금액 확정 한 번, 이미 기록한 사건은 중복 반영하지 않음 |
| `burl` 자체 유실·미호출 | 5초 만료 뒤 잠재 지출 반환, 광고주 지출 없음 |
| 처리 주체·인스턴스·배치 변경 뒤 기존 예약의 `burl` | 예약 자체가 5초 안이면 금액 확정 한 번 |
| DSP 장애로 예약 사용량 불명확 | 잠재 지출 가능 시간이 끝날 때까지 불명확한 금액을 재할당하지 않음 |
| 5초 뒤 `burl` | 기록만 남고 금액 변화 없음 |
| 모순된 `lurl`과 `burl` | 격리, 자동 금액 재변경 없음 |
| 렌더링 통지 2초 경계 | SSP의 복구 가능한 최초 접수 시각으로 판정 |
| 잘못 연결·중복·변조된 렌더링 통지 | 과금 거부 |
| 외부 DSP 오류·시간 초과 | 다른 DSP로 경매 지속, p99 50ms |
| 내부 저장 경로 불확실 | 성공·광고 없음으로 숨기지 않고 기술 실패 |
| DSP 과금 통지 접수 경로 적체 | 예산 안전성과 정한 미과금 위험 상한 중 하나라도 지키지 못하면 프로젝트 DSP의 새 입찰 중지, 외부 DSP 경매 지속 |

정상 부하에서는 모순된 결과를 만들지 않는다. 계약 충돌은 별도 결함 시험에서만 주입한다.

## 6. 배포·부하 발생 조건

| 항목 | 기준 |
|---|---|
| 클라우드 | AWS 리전 2곳, 한 곳은 `ap-northeast-2`, 시험 후 제거 |
| 장애 영역 | 서로 다른 리전 2곳, 인스턴스·AZ·리전 장애를 각각 주입 |
| 비교 단위 | Linux x86_64, 2 vCPU, 8GiB |
| 부하 발생기 | 대상과 분리된 호스트 |
| 공급자 역할 | k6와 통합 시험 |
| 네트워크 | 진입 네트워크와 리전 내부·리전 간 경로를 구분해 기록 |
| 연결 | 준비된 재사용 연결, 최초 DNS·연결 수립 제외 |
| 초기 통신 | HTTP POST + JSON, 암호화 사용 |

k6는 `constant-arrival-rate`를 사용하고 `dropped_iterations=0`이어야 한다. 목표 요청 수와 실제 요청 수가 같아야 하며 `http_req_blocked`, 연결, 송신 시간과 부하 발생기의 CPU·메모리·네트워크를 따로 남긴다. HTTP 호출 전후의 호출자 전체 시간과 k6의 `http_req_duration`을 함께 기록한다. 부하 발생기 증설로 결과가 유의미하게 바뀌면 기존 시험은 무효다.
