# 부하·데이터·검증 기준

상태: 기준선 2.0

상위 문서: [제품·도메인 요구사항](product-domain-requirements.md)

이 문서는 어떤 부하, 데이터, 장애에서 무엇을 통과로 볼지 정한다. 복제본 수, 부하 분산, 저장소는 정하지 않는다.

## 1. 기준의 성격

- 일 1천만 경매는 특정 회사의 실측치가 아니라 포트폴리오 사업 가정이다.
- 일일 물량의 50%가 가장 바쁜 4시간에 모인다는 것도 재현을 위한 가정이다.
- 50ms는 정상 경로의 내부 목표고, 1,000 RPS는 예상 정점이 아니라 순간 부하·회복 시험이다.
- OpenRTB는 CPM과 낙찰·과금 통지를 구분한다. [IAB OpenRTB 2.6](https://github.com/InteractiveAdvertisingBureau/openrtb2.x/blob/main/2.6.md#423---object-bid)
- 180ms는 국내 S2S 경매 제한 시간의 참고값이다. [Kakao AdX 입찰 요청 규격](https://kakaobusiness.gitbook.io/main/partner/kakaoadx_rtb/spec#object-bid-request)

## 2. 부하

```text
일평균 = 10,000,000 / 86,400 = 115.74 RPS
바쁜 시간 평균 = 5,000,000 / 14,400 = 347.22 RPS
설계 정점 = 400 RPS
```

| 구분 | 공급자 요청률 | 지속 시간 | 의미 |
|---|---:|---:|---|
| 일평균 | 115.74 RPS | 24시간 환산 | 사업 물량 |
| 설계 정점 | 400 RPS | 바쁜 4시간 | 예상 수요 |
| 정상 합격 | 500 RPS | 10분 | 정점 대비 25% 여유 |
| 순간 부하 | 1,000 RPS | 60초 | 정확성과 회복 검증 |

경매당 자사 DSP 1개와 외부 DSP 2개를 호출한다.

| 공급자 RPS | SSP의 DSP 호출/초 | 자사 DSP 요청/초 |
|---:|---:|---:|
| 400 | 1,200 | 400 |
| 500 | 1,500 | 500 |
| 1,000 | 3,000 | 1,000 |

500 RPS에서 자사 DSP는 초당 캠페인 검색 500건, 잠재 지출 확보 약 300건, 낙찰 100건, 과금 90건을 처리한다.

## 3. 데이터

| 데이터 | 규모·분포 |
|---|---|
| 공급자 | 1,000개 |
| 게재 위치 | 10,000개, 상위 20%가 요청 80% 처리 |
| 광고주 | 10,000개, KRW |
| 캠페인 | 100,000개, 활성 30,000개 |
| 소재 | 200,000개, 캠페인당 평균 2개 |
| 활성 후보 | 요청당 p50 20 / p95 200 / p99 1,000 |
| 예산 편향 | 상위 1% 광고주에 금액 연산의 20% 집중 |
| 소진 임박 | 광고주·활성 캠페인 1%는 노출 1~10건 금액만 남김 |

입찰 CPM은 500, 1,000, 1,500, 2,000, 3,000 KRW를 사용한다. 광고주 충전액은 100만~1억 KRW, 캠페인 일일 한도는 10만~500만 KRW다. 잠재 지출은 5분 뒤 만료한다.

## 4. 요청과 장애 구성

정상 요청은 배너 300×250, 광고 영역 1개, KRW, 1차 가격이다. 자사 DSP는 60%에 입찰하고 전체 경매의 20%를 낙찰한다. 낙찰의 90%가 과금되고 10%는 만료된다.

| 장애 | 주입 | 기대 결과 |
|---|---:|---|
| 중복 낙찰·과금·패찰 | 해당 이벤트의 5% | 금액 효과 1회 |
| 과금 후 낙찰 도착 | 자사 낙찰의 1% | 동일한 최종 상태 |
| 패찰·과금 충돌 | 입찰의 0.1% | `MONEY_EVENT_CONFLICT` |
| 종료 이벤트 누락 | 자사 낙찰의 10% | 5분 후 잠재 지출 해제 |
| 외부 DSP 오류 | 요청의 5% | 다른 DSP로 경매 지속 |
| 외부 DSP 시간 초과 | `tmax + 300ms` 후 응답 | 늦은 입찰 제외 |
| 잘못된 입찰 | 외부 DSP 입찰의 5% | 낙찰 후보에서 제외 |

## 5. 목표

| 구분 | 목표 |
|---|---|
| 경매 제한 시간 | 기본 `tmax=180ms` |
| 정상 경로 | 500 RPS에서 p99 ≤ 50ms |
| DSP 시간 초과 | 400 RPS에서 p99 ≤ 200ms |
| 순간 부하 | 1,000 RPS에서 p99 ≤ 180ms |
| SSP 인스턴스 중단 | 전체 중단 0, 10초 안에 성공률 99.9% 회복 |
| 자사 DSP 인스턴스 중단 | 전체 중단 0, 10초 안에 성공률 99.9% 회복 |
| AZ 중단 | 전체 중단 0, 30초 안에 성공률 99.9% 회복 |
| 인정된 금액 효과 | RPO 0, 유실·중복 0건 |
| 과부하 회복 | 부하 감소 후 30초 안에 정상 범위 회복 |

전체 중단 0은 장애 구간의 어느 1초 구간도 성공률 0%가 되지 않는다는 뜻이다. 장애 순간의 처리 중 요청 일부는 실패할 수 있다.

## 6. 검증 환경

| 항목 | 기준 |
|---|---|
| 클라우드 | AWS `ap-northeast-2`, 시험 후 제거 |
| 장애 영역 | 최소 2개 AZ |
| 애플리케이션 비교 단위 | Linux x86_64, 2 vCPU, 8GiB |
| 부하 발생기 | 대상과 분리된 호스트 |
| 네트워크 | 사설 IP, AZ 간 경로 포함, 첫 버전에서 TLS 비용 제외 |
| 초기 통신 | HTTP POST + JSON |

정확한 복제본, 부하 분산, 저장소, 복제 구조는 ADR 후에 확정한다.
