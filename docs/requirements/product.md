# 제품·도메인 요구사항

상태: 확정안

수치와 합격 기준: [부하·데이터·검증 기준](workload.md)

이 프로젝트는 웹 배너 RTB에서 제한 시간 경매, 캠페인 선택과 페이싱, 분산 예산 정합성, 장애 지속성을 구현·검증한다.

## 1. 범위

| 역할 | 책임 |
|---|---|
| 공급자 | 광고 기회를 요청하고 렌더링 결과를 SSP에 통지한다. k6와 통합 시험이 모의한다. |
| SSP | 요청 검증, DSP 호출, 슬롯별 낙찰, 렌더링 검증과 과금 가능 사건의 통지를 담당한다. |
| 프로젝트 DSP | 캠페인·소재·예산으로 입찰하고 잠재 지출을 확정·해제·만료한다. |
| 외부 DSP | 서로 다른 회사를 나타내는 독립 모의 시스템이다. |
| 광고주 | 프로젝트 DSP에 캠페인과 총예산을 맡긴다. |

공급자, SSP, 프로젝트 DSP, 두 외부 DSP는 서로 다른 회사다. SSP에서 보면 셋 모두 독립된 DSP 회사이며 같은 경매·통지 계약을 따른다. 프로젝트 DSP만 내부를 제품 수준으로 구현하며, 여러 인스턴스가 하나의 논리적 캠페인 예산을 공유하고 게이트웨이 뒤에서 동작한다. 외부 DSP는 각 회사의 독립 모의 주소이며 이 게이트웨이를 사용하지 않는다.

SSP는 경매 결과와 과금 가능한 렌더링 사건의 권위자이고, 프로젝트 DSP는 캠페인과 광고주 예산의 권위자다. 두 회사는 독립 배포·독립 저장소를 사용하며 상대 저장소나 내부 원장을 호출하지 않는다. SSP의 과금 가능 사건, 프로젝트 DSP의 광고주 캠페인 차감, 두 회사 사이의 지급·정산은 서로 다른 사실이다.

SSP와 프로젝트 DSP의 핵심 경로는 제품 수준으로 구현한다. 공급자 애플리케이션, 광고 표시 SDK, 실제 브라우저, 외부 DSP 내부는 구현하지 않는다.

다음은 범위 밖이다.

- 클릭·기여 추적, 사용자 식별·프로파일링, 세부 타기팅
- 캠페인 관리 화면과 실행 중 예산·입찰가·기간 변경
- 머신러닝 입찰 최적화, 성과 예측, 고급 페이싱, 빈도 제한
- 광고 사기·가시성·소재 심사·브랜드 안전성
- 복수 통화, 충전·환불·청구서·업체 간 회계 대사
- 쿠버네티스, 자동 확장 제어면

## 2. 요구사항

### `요구-01` 제한 시간 경매

SSP는 요청을 검증하고 프로젝트 DSP와 두 외부 DSP에 의미상 같은 OpenRTB 2.6 요청을 병렬로 보낸다. DSP별 결과는 `VALID_BID`, `NO_BID`, `TIMEOUT`, `INVALID_BID`, `ERROR` 중 하나로 끝난다.

- 요청은 하나 이상의 광고 슬롯을 가질 수 있고 각 슬롯은 독립된 경매·낙찰·과금 단위다.
- `입찰 마감 = SSP 요청 수신 시각 + 유효 tmax`이며 마감 뒤 응답은 결과를 바꾸지 않는다.
- 슬롯별 유효 입찰 중 최저가 이상인 최고 CPM이 낙찰한다.
- 1가격 경매를 사용하며 낙찰 CPM은 DSP가 제출한 CPM과 같다.
- 최고가 동률은 도착 순서나 고정 DSP 우선순위가 아니라 경매·슬롯 식별자를 이용해 결정적으로 분산한다.
- 한 DSP가 실패해도 다른 유효 입찰로 경매를 끝내며 유효 입찰이 없으면 정상적인 광고 없음으로 응답한다.
- 동일한 `auctionId + impressionId`는 최대 한 낙찰과 한 과금 결과만 만든다.
- 중복 공급자 요청은 새 캠페인 선택·잠재 지출·낙찰·과금을 만들지 않는다.

### `요구-02` 캠페인 입찰과 생명주기

프로젝트 DSP는 슬롯마다 최대 한 캠페인과 소재를 선택해 하나의 입찰을 반환한다.

입찰 후보는 다음 조건을 모두 만족해야 한다.

- 현재 시각이 캠페인의 시작·종료 기간 안이고 캠페인이 활성 상태다.
- 소재의 형식과 크기가 슬롯에 맞는다.
- 입찰 금액의 잠재 지출을 만들 수 있다.
- 기본 페이싱 정책이 현재 입찰을 허용한다.

후보가 여러 개면 페이싱 목표보다 가장 뒤처진 캠페인을 선택하고, 같은 상태면 `campaignId` 순으로 결정한다. 적격 후보가 없으면 `NO_BID`한다. 이용자·지역·행동·매체 분류 타기팅은 다루지 않는다.

캠페인은 시험 전에 식별자, 총예산, 고정 입찰 CPM, 시작·종료 시각, 소재와 함께 등록한다. 일시 중지와 재개는 허용하지만 실행 중 다른 값은 변경하지 않는다. 중지·종료 뒤 새 입찰은 만들지 않으며 이미 만든 잠재 지출은 원래 규칙대로 종결한다.

동일한 DSP 요청을 다시 받으면 최초 캠페인·소재·가격·`bidId`와 잠재 지출을 재사용한다. 최초 결과를 안전하게 확인할 수 없으면 새 입찰을 만들지 않고 `NO_BID`한다.

### `요구-03` 가격과 캠페인 예산

통화는 KRW 하나이며 입찰가와 최저가는 CPM이다.

- CPM은 소수점 셋째 자리까지 허용하고 초과 정밀도는 거부한다.
- 노출 1건 금액은 `CPM / 1,000`으로 정확하게 보존한다.
- 입찰·잠재 지출·과금 과정에서 반올림하지 않는다.
- 내부 자료형과 실제 대금 정산의 반올림은 이후 결정 범위다.

캠페인에는 하나의 총예산만 있다.

```text
캠페인 확정 지출 + 유효한 잠재 지출 <= 캠페인 총예산
```

프로젝트 DSP는 입찰을 반환하기 전에 슬롯별 입찰 금액의 잠재 지출을 만든다. 한 캠페인이 같은 요청의 여러 슬롯에 입찰하면 모든 잠재 지출의 합을 검사한다. 예산이 정확히 소진되는 것은 허용하지만 초과 지출은 허용하지 않는다.

| 사건 | 금액 효과 |
|---|---|
| 입찰 제시 | DSP 예약 시각부터 노출 1건 금액을 잠재 지출로 잡는다. |
| `nurl` | 잠재 지출을 유지한다. |
| `lurl` | 만료 전에 접수하면 잠재 지출을 해제한다. |
| `burl` | 만료 전에 복구 가능하게 접수하면 잠재 지출을 확정 지출로 바꾼다. |
| 처리 주체·인스턴스·배치 변경 | 새 예약의 처리 주체는 바뀔 수 있지만 이미 만든 잠재 지출은 바꾸지 않는다. |
| 만료 | 다른 결과 없이 예약 후 95초가 지나면 잠재 지출을 해제한다. |
| 늦은 `burl` | 기록만 남기고 금액을 바꾸지 않는다. |

95초는 렌더링 가능 시간 60초, 통지 전달·장애 복구 여유 30초, 정상 통지 접수 목표 5초의 합이다. DSP가 복구 가능하게 접수한 시각으로 기한을 판정한다. 입찰 응답이 SSP에 도착하지 않은 잠재 지출도 같은 기한 안에 끝난다.

같은 사건은 여러 번 전달되어도 금액에 한 번만 영향을 준다. 모순된 `lurl`과 `burl`은 정상 입력이 아니라 계약 결함이며 DSP는 금액을 다시 변경하지 않고 격리한다.

예약 처리 주체·인스턴스·배치가 바뀌어도 이미 만든 잠재 지출은 `lurl`, `burl`, 95초 만료 중 하나로 끝날 때까지 캠페인 예산을 계속 점유한다. 장애로 사용 내역을 확신할 수 없으면 초과 지출을 막기 위해 사용 가능성이 있는 금액을 보수적으로 점유하며, 미사용임을 복구 가능하게 증명하거나 잠재 지출 가능 시간이 끝난 뒤에만 가용 예산으로 반환한다.

95초 뒤의 `burl`은 광고주 캠페인 금액을 바꾸지 않는다. 이때 SSP와 프로젝트 DSP 사이에 발생할 수 있는 지급·손실과 회계 대사는 구현 범위 밖이며, 광고주 예산 원장과 섞지 않는다.

### `요구-04` 기본 페이싱

캠페인 총예산을 초과하지 않으면서 시작부터 종료까지 지나치게 일찍 소진되지 않게 한다.

- 기본 목표는 기간에 비례하는 선형 누적 지출이다.
- 목표보다 앞선 캠페인의 입찰 참여를 줄이고 뒤처진 캠페인을 우선한다.
- 적격 광고 기회가 충분하면 기간 전체에 걸쳐 예산을 소비해야 한다.
- 적격 광고 기회가 부족한 경우 미소진은 허용한다.
- 허용 편차, 계산 주기, 잠재 지출 반영 비율과 분산 제어 방식은 검증 기준과 ADR에서 구체화한다.

### `요구-05` 렌더링과 통지

DSP는 입찰 응답에 사전 등록된 `nurl`, `lurl`, `burl`을 제공한다. SSP는 경매 응답 경로 밖에서 비동기로 호출한다.

- SSP가 낙찰을 결정하면 `nurl`, 패찰을 결정하면 `lurl`을 보낸다.
- 렌더링 가능 시간은 SSP 낙찰 결정부터 60초다.
- SSP가 60초 안에 렌더링 통지를 복구 가능하게 접수하면 자신의 계약상 과금 가능 사건을 확정하고 `burl`을 보낸다.
- DSP가 예약 후 95초 안에 `burl`을 복구 가능하게 접수해야 캠페인 과금이 확정된다.
- 렌더링되지 않은 낙찰을 위한 별도 미과금 통지는 보내지 않는다.
- SSP의 렌더링 접수 성공은 SSP가 과금 가능 사건과 전달 책임을 복구할 수 있다는 뜻이다. DSP의 `burl` 접수 성공은 DSP가 캠페인 금액 효과를 복구할 수 있다는 뜻이다. 두 성공을 하나의 분산 트랜잭션으로 묶지 않는다.
- 정상 상황에서 SSP의 통지 생성부터 DSP의 복구 가능한 접수까지 p99는 5초 이하다.

SSP의 접수 시각으로 60초 경계를 판정하며 클라이언트가 주장한 시각은 관측에만 쓴다. 외부 입력의 가격·캠페인·과금 주장을 믿지 않고 SSP가 실제로 만든 낙찰 결과와 연결된 통지만 인정한다. 중복·변조·재사용된 통지는 추가 과금을 만들 수 없다. 연결과 상대 인증의 구현 방식은 ADR에서 정한다.

### `요구-06` 부분 실패와 과부하

- 외부 DSP 하나의 실패는 해당 입찰에만 영향을 주고 SSP 경매는 계속한다.
- 프로젝트 DSP가 예산을 안전하게 예약할 수 없으면 기술 실패 또는 `NO_BID`로 끝내고 예산을 추측하지 않는다.
- SSP가 경매 결과나 렌더링 통지를 복구 가능하게 보존할 수 없으면 성공으로 응답하지 않는다.
- 프로젝트 DSP의 통지 접수가 일시적으로 불가능해도 안전하게 증명할 수 있는 예약 한도가 있고 SSP가 과금 가능 사건을 복구 가능하게 보존하며 재시도 적체가 상한 안이면 그 범위에서 새 입찰을 계속할 수 있다. 프로젝트 DSP는 어느 조건이라도 만족하지 못하면 새 입찰을 중단한다. SSP는 가능한 경우 외부 DSP와 경매를 계속한다.
- 처리 능력을 넘은 요청은 광고 없음으로 숨기지 않고 명시적인 기술 실패로 빠르게 거절한다.
- 과부하에서도 잘못된 낙찰, 예산 초과, 중복 과금과 성공으로 인정한 사실의 유실은 허용하지 않는다.

### `요구-07` 장애 지속성과 상태 보존

SSP 회사와 프로젝트 DSP 회사는 각각 서로 다른 AWS 리전 두 곳에 여러 인스턴스를 둔다. 프로젝트 DSP 게이트웨이와 예산 원장도 인스턴스·AZ·리전 하나가 전역 단일 장애점이 되면 안 된다.

- 인스턴스·AZ·리전 하나가 중단되어도 다른 리전은 경매를 계속하고, 프로젝트 DSP는 안전하게 증명할 수 있는 예산 권한 안에서 입찰을 계속한다.
- 장애 영역에만 속한 캠페인·예산 권한의 일시적 사용 중단은 허용하지만 다른 캠페인과 외부 DSP까지 중단시키지 않는다.
- SSP가 성공으로 인정한 경매 결과·렌더링 접수·통지 전달 상태의 RPO는 0이다.
- 프로젝트 DSP가 성공으로 인정한 통지 접수·캠페인 금액 효과의 RPO는 0이다.
- 진행 중 경매 계산은 실패할 수 있지만 이미 생성한 잠재 지출은 95초 안에 종결한다.
- 복구 뒤 중복 전달이 발생해도 낙찰과 금액 효과는 바뀌지 않는다.
- SSP와 DSP는 상대 저장소나 내부 장부를 조회·호출하지 않는다. 응답 유실은 공개 계약 주소에 같은 사건을 멱등 재시도하여 해소한다.
- 리전 장애 중 사용 여부를 증명할 수 없는 예산은 다른 리전에 즉시 재할당하지 않는다. 복구하거나 최대 잠재 지출 기한이 끝날 때까지 보수적으로 점유한다.

### `요구-08` 검증과 통신 독립성

경매, 캠페인 선택, 페이싱, 금액 상태, 자원 포화와 장애 전환을 연결해 관측하고 같은 조건으로 재현할 수 있어야 한다. JSON을 이진 표현으로 바꾸어도 경매·금액 의미와 합격 기준은 달라지지 않는다.

## 3. 제약과 가정

- SSP–DSP 계약은 OpenRTB 2.6의 명시적 하위 규격을 따른다.
- 공급자 API는 프로젝트 전용이며 OpenRTB 객체를 직접 노출하지 않는다.
- 웹 배너, KRW, 1가격 경매를 우선한다.
- 실행 환경은 Java 21과 서로 다른 AWS 리전 두 곳이다. 한 곳은 서울 리전이며 다른 리전의 선택은 배포 결정에서 정한다.
- HTTP/JSON은 초기 표현일 뿐 영구 제약이 아니다.
- 실제 사람이 광고를 보았는지 판별하는 광고 사기 탐지는 하지 않는다. 유효한 낙찰과 연결된 클라이언트 통지를 렌더링 성공으로 간주한다.
- 구현이 어렵다는 이유로 예산 불변식, 50ms 목표 또는 장애 범위를 낮추지 않는다. 요구를 바꾸려면 ADR이 아니라 이 문서부터 바꾼다.
