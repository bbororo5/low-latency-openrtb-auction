# 아키텍처 중요 요구사항

상태: 확정안

근거: [제품·도메인 요구사항](../requirements/product-domain-requirements.md), [부하·데이터·검증 기준](../requirements/workload-data-verification.md)

## 1. 목록

| ID | 요구사항 | 우선순위 |
|---|---|---|
| `ASR-01` | 정상 첨두에서 정확한 경매를 p99 50ms 안에 끝낸다. | 최우선 |
| `ASR-02` | 실패한 DSP와 초과 부하를 격리하고 보호 처리량을 유지한다. | 최우선 |
| `ASR-03` | 분산 인스턴스에서도 캠페인 예산을 초과하지 않고 모든 잠재 지출을 95초 안에 끝낸다. | 최우선 |
| `ASR-04` | 총예산을 캠페인 기간에 걸쳐 소비하도록 기본 페이싱을 제공한다. | 중요 |
| `ASR-05` | 인스턴스·게이트웨이·AZ 장애에도 서비스와 인정된 사실을 보존한다. | 최우선 |
| `ASR-06` | 신뢰하지 않는 외부 입력이 낙찰과 과금을 위조·중복하지 못하게 한다. | 최우선 |
| `ASR-07` | 성능·금액·페이싱·장애 결과를 재현하고 설명할 수 있다. | 중요 |
| `ASR-08` | 통신 표현과 배포 구조가 바뀌어도 경매와 금액 의미를 유지한다. | 중요 |

## 2. 시나리오

### `ASR-01` 정상 경로

- **상황:** 요청당 p99 1,000개 활성 후보가 있는 데이터에서 500 RPS가 10분간 들어온다.
- **응답:** 자사 DSP가 캠페인 예산을 예약해 입찰하고 SSP가 세 DSP의 유효 입찰로 슬롯별 1가격 낙찰을 결정한다.
- **측정:** 기술 실패·누락·잘못된 낙찰 0건, 호출자 관측 p99 50ms 이하, 180ms 뒤 결과 0건.
- **압력:** 진입 네트워크, 병렬 외부 호출, 후보 검색과 금액 예약이 하나의 시간 예산을 공유한다.

### `ASR-02` 실패와 과부하 격리

- **상황:** 500 RPS에서 외부 DSP 하나가 실패하거나 1,000 RPS가 60초간 유입된다.
- **응답:** 실패한 DSP만 제외한다. 과부하에서는 최소 400 RPS를 정상 처리하고 초과분을 기술 실패로 빠르게 거절한다.
- **측정:** 보호 요청 p99 50ms 이하, 잘못된 낙찰·금액 위반 0건, 부하 감소 뒤 30초 안에 회복.
- **압력:** 시간 제한, 자원 상한, 격리와 부하 차단이 없으면 하나의 지연이 전체 경매로 전파된다.

### `ASR-03` 예산과 잠재 지출

- **상황:** 여러 DSP 인스턴스가 같은 캠페인 예산에 동시에 입찰하고 응답·`lurl`·`burl`이 중복·지연·유실된다.
- **응답:** 슬롯별 입찰 전에 잠재 지출을 만들고, 만료 전 `lurl`은 해제하며 `burl`은 확정 지출로 바꾼다. 결과가 없으면 DSP 예약 시각부터 95초에 해제하고 늦은 `burl`은 금액에 반영하지 않는다.
- **측정:** `확정 지출 + 유효한 잠재 지출 <= 총예산`, 초과 지출·중복 과금·95초 뒤 고아 잠재 지출 0건.
- **압력:** 빠른 분산 예약, 중복 처리, 장애 복구와 시간 기반 종료를 동시에 만족해야 한다.

### `ASR-04` 기본 페이싱

- **상황:** 페이싱이 없으면 기간의 25% 안에 예산을 소진할 만큼 적격 요청이 충분히 들어온다.
- **응답:** 기간에 비례한 선형 목표보다 앞선 캠페인의 입찰을 줄이고 뒤처진 캠페인을 우선한다.
- **측정:** 시험 전에 산정한 허용 편차 안에서 기간에 걸쳐 소비하고 총예산을 넘지 않는다. 적격 요청 부족에 따른 미소진은 실패가 아니다.
- **압력:** 확정 지출, 진행 중 잠재 지출, 분산 상태 가시성 지연을 이용해 입찰을 제어해야 한다.

### `ASR-05` 장애 지속성과 보존

- **상황:** 400 RPS에서 SSP·자사 DSP·게이트웨이 인스턴스 하나 또는 AZ 하나가 중단되거나 순차 배포된다.
- **응답:** 다른 인스턴스와 AZ가 처리를 이어가며 성공으로 인정한 경매 결과, 렌더링·통지 접수와 금액 효과를 복구한다.
- **측정:** 어느 1초 구간도 성공률 0%가 아니며 인스턴스 장애는 10초, AZ 장애는 30초 안에 성공률 99.9%·p99 50ms로 회복한다. 인정된 사실의 RPO는 0이다.
- **압력:** 실행 중 계산은 버릴 수 있지만 외부에 성공을 알린 사실과 금액은 인스턴스 메모리와 함께 사라지면 안 된다.

### `ASR-06` 신뢰 경계

- **상황:** 외부 주체가 잘못 연결·변조·중복된 입찰 또는 렌더링 통지를 보낸다.
- **응답:** 상대와 권한을 확인하고 SSP가 만든 요청·슬롯·낙찰 결과·60초 기한과 연결되는 사건만 인정한다. 외부가 주장한 가격과 캠페인 값은 금액 원본으로 쓰지 않는다.
- **측정:** 잘못된 낙찰, 위조·중복 과금과 기한 뒤 렌더링 인정 0건.
- **압력:** 구현하지 않는 클라이언트와 다른 회사 시스템이 금액을 발생시키는 경계를 통과한다.

### `ASR-07` 검증 가능성

- **상황:** 용량, 페이싱, 통합, 과부하와 인스턴스·AZ 장애 시험을 반복한다.
- **응답:** 경매·캠페인·예약·통지·금액 상태와 자원 포화·장애 전환을 같은 식별자로 연결해 관측한다.
- **측정:** 동일 조건을 재현하고 실패 원인, 회복 시각, 예산 불변식과 페이싱 편차를 설명할 수 있다.
- **압력:** 분산 흐름의 결과만 모으면 지연과 금액 오류가 발생한 경계를 판별할 수 없다.

### `ASR-08` 변경 가능성

- **상황:** HTTP/JSON을 이진 표현으로 바꾸거나 복제본과 배포 버전을 변경한다.
- **응답:** 1가격 낙찰, 가격 정밀도, 금액 사건과 기한의 의미를 유지한다.
- **측정:** 기존 도메인·성능·금액 시험을 그대로 통과하고 순차 배포 중 상태 손상 0건.
- **압력:** 직렬화·저장·배포 선택이 도메인 계약으로 새어 들어오면 최적화와 진화가 어려워진다.

## 3. 우선순위

1. 잘못된 낙찰과 금액 위반을 막는 것이 처리량보다 우선한다.
2. 제한 시간을 넘긴 입찰은 가격이 높아도 낙찰시키지 않는다.
3. 저장 결과를 확신할 수 없으면 성공·광고 없음·정상 무입찰로 숨기지 않는다.
4. 과부하에서는 정확성과 지연을 보호하기 위해 초과 요청과 자사 입찰 기회를 포기할 수 있다.
5. 95초가 지나면 가용성을 위해 금액 정합성을 포기하는 것이 아니라, 합의한 계약에 따라 예약을 해제하고 늦은 과금을 거부한다.
