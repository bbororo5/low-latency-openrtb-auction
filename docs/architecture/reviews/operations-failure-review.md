# 운영과 실패 관점 검토

검토 대상: [아키텍처 동인](../architecture-drivers.md)

## 결론

인스턴스와 AZ 중단만으로는 실패 범위가 부족하다. 실제로는 완전히 중단된 상태보다 느린 응답, 부분 단절, 자원 고갈, 재시도 폭증처럼 살아 있지만 처리할 수 없는 상태가 더 오래 장애를 전파한다. `동인-03`과 `동인-04`는 이러한 실패와 회복까지 포함해야 한다.

## 실패별 기대 동작

| 실패 | 허용되는 영향 | 허용되지 않는 영향 |
|---|---|---|
| 외부 DSP 지연·오류 | 해당 DSP의 입찰 제외 | 다른 DSP와 경매 전체의 실패 |
| SSP·자사 DSP 인스턴스 중단 | 처리 중 요청 일부 실패 | 어느 1초 구간의 성공률 0%, 금액 불일치 |
| AZ 단절 | 해당 AZ 처리 감소와 일시적 성공률 저하 | 전체 중단, 인정된 금액 효과 유실·중복 |
| 연결·스레드·대기열 고갈 | 제한된 요청 거절 또는 무입찰 | 무한 대기와 부하 감소 뒤 장기 지연 |
| 저장 경로 지연·전환 | 금액 변경이 필요한 입찰의 실패 | 오래된 예산으로 입찰하거나 성공 응답 뒤 금액 유실 |
| 재시도 폭증 | 제한된 재시도와 중복 입력 | 원래 부하보다 큰 증폭과 중복 과금 |
| 순차 배포 | 일부 인스턴스 교체 | 전체 중단과 버전 차이에 의한 상태 손상 |

## 우아한 성능 저하 순서

1. 잘못된 낙찰과 금액 위반을 막는다.
2. 제한 시간을 넘길 작업을 더 받거나 기다리지 않는다.
3. 실패한 DSP와 인스턴스의 영향을 해당 경계 안에 가둔다.
4. 처리 능력을 넘으면 제한된 범위에서 요청을 거절한다.
5. 부하가 줄면 재시작 없이 정상 범위로 돌아온다.
6. 복구 후 금액 상태를 대조해 유실과 중복이 없음을 확인한다.

우아한 성능 저하는 모든 요청 성공이 아니라 핵심 불변식을 지키면서 실패 범위를 제한하는 것이다.

## 운영 판정 기준

- 정상 부하, 순간 부하, 의존성 지연을 서로 구분해 측정한다.
- 처리량과 응답 시간뿐 아니라 연결, 스레드, 대기열, 처리 중 작업을 함께 본다.
- 장애 감지, 트래픽 제외, 복구 시작, 성공률 회복 시각을 각각 기록한다.
- 성공률이 회복되어도 밀린 작업과 자원 사용량이 정상화되지 않으면 복구로 보지 않는다.
- 장애 시험 전후의 잠재·확정 지출을 대조한다.
- 동일 장애를 반복 주입해 복구 시간이 합격 기준 안에 드는지 확인한다.

## 동인 수정 사항

- `동인-03`은 단순한 복제보다 장애 감지부터 트래픽 전환, 상태 복구, 금액 대조까지의 연속 과정으로 표현해야 한다.
- `동인-04`는 외부 DSP 실패뿐 아니라 내부 자원 포화와 재시도 증폭을 포함해야 한다.
- 금액 경로가 불확실할 때 성공을 반환하지 않는 원칙을 우선순위에 추가해야 한다.
- 관측 항목은 장애 원인과 회복 여부를 판정할 수 있도록 자원 상한과 상태 전환을 포함해야 한다.

## ADR로 넘길 질문

- 각 외부 호출과 경매 전체에 시간 예산을 어떻게 나눌 것인가?
- 연결·스레드·대기열·처리 중 작업에 어떤 상한과 거절 규칙을 둘 것인가?
- 재시도할 실패와 즉시 끝낼 실패를 어떻게 구분하고 증폭을 어떻게 막을 것인가?
- 인스턴스와 AZ 장애를 어떤 신호로 감지하고 언제 트래픽에서 제외할 것인가?
- 금액 저장 경로가 느리거나 전환 중일 때 입찰과 과금 요청을 어떻게 처리할 것인가?
- 복구 완료와 서비스 정상화를 어떤 지표 조합으로 판정할 것인가?
