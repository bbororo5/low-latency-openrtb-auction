# 신뢰 경계 관점 검토

검토 대상: [아키텍처 동인](../architecture-drivers.md)

## 결론

공급자·렌더링 클라이언트와 외부 DSP는 신뢰 경계 밖에 있다. 상대를 인증해도 그 내용이 올바르다는 뜻은 아니다. SSP가 자신이 만든 요청·낙찰과 연결하고 기한·중복을 검증한 뒤에만 경매와 과금 후보가 된다.

## 사실의 권한

| 주체 | 주장할 수 있는 사실 | 확정할 수 없는 사실 |
|---|---|---|
| 렌더링 클라이언트 | 광고가 렌더링됐다는 주장 | 낙찰가와 캠페인 과금 |
| 외부 DSP | 슬롯에 대한 입찰 | 입찰 유효성·최종 낙찰 |
| SSP | 유효 입찰·낙찰·60초 안의 렌더링과 과금 후보 | 자사 DSP 캠페인 금액 변경 |
| 자사 DSP 입찰 | 캠페인·소재·입찰가 | SSP의 최종 낙찰 |
| 자사 DSP 예산 | 잠재·확정 지출과 만료 | 렌더링 사실 자체 |

## 경계별 검증

| 경계 | 검증할 것 |
|---|---|
| 공급자 → SSP | 등록된 상대와 요청 권한 |
| 렌더링 클라이언트 → SSP | 실제 낙찰·슬롯 연결, 60초 기한, 중복·변조·재사용 |
| DSP → SSP | 요청·슬롯 연결, 180ms 마감, 가격 정밀도, 통화와 소재 |
| SSP → 자사 DSP | 등록된 상대, 예약·입찰 연결, 사건 종류, 95초 기한과 중복 |

외부 입력의 가격·캠페인·시각을 금액 원본으로 사용하지 않는다. 실제 사람이 광고를 보았다는 증명과 광고 사기 탐지는 범위 밖이며, 유효한 낙찰에 연결된 클라이언트 통지를 렌더링 성공으로 간주한다. 이를 증명하는 토큰·서명·서버 상태 방식은 아직 선택하지 않는다.

## ADR로 넘길 질문

- 서버 간 상대와 권한을 어떻게 확인할 것인가?
- 렌더링 통지를 실제 낙찰과 어떤 정보로 연결할 것인가?
- 변조·재사용·중복을 어떤 식별자와 보존 기간으로 판정할 것인가?
- 사전 등록된 DSP 통지 주소를 어떻게 제한할 것인가?
