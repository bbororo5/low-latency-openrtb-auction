# 신뢰 경계 관점 검토

검토 대상: [아키텍처 동인](../architecture-drivers.md)

## 결론

현재 동인은 외부 입력의 오류는 다루지만 **누가 낙찰과 과금을 확정할 권한이 있는지**를 명시하지 않았다. 클라이언트 렌더링 통지와 외부 DSP 입찰은 주장일 뿐이며, 각 책임자가 식별·연결·중복·권한을 검증한 뒤에만 경매와 금액 사실이 된다.

## 사실의 권한

| 주체 | 주장할 수 있는 사실 | 확정할 수 없는 사실 |
|---|---|---|
| 클라이언트 | 낙찰 광고가 렌더링됐다는 통지 | 과금 확정, 낙찰가, 광고주 예산 변경 |
| 공급자 | 자신의 광고 영역에서 검증한 과금 가능 노출 | SSP의 낙찰 결과, 자사 DSP의 금액 상태 |
| 외부 DSP | 특정 광고 영역에 대한 입찰 | 입찰 유효성, 최종 낙찰, 공급자 노출 |
| SSP | 유효 입찰, 최종 낙찰, 공급자 통지와 낙찰의 연결 | 자사 DSP의 캠페인·예산 변경 |
| 자사 DSP 입찰 | 캠페인·소재 선택과 입찰가 | SSP의 최종 낙찰과 공급자 노출 판정 |
| 자사 DSP 예산 | 잠재 지출과 확정 지출의 변경 | 경매 결과와 렌더링 사실 자체 |
| 캠페인 원본 시스템 | 캠페인·소재·한도 원본과 버전 | 실행 시점의 입찰·낙찰 결과 |

## 경계별 검증

| 경계 | 검증할 것 |
|---|---|
| 클라이언트 → 공급자 | 발급된 광고와의 연결, 중복, 허용 시간, 조작 여부 |
| 공급자 → SSP | 공급자와 광고 영역 권한, 경매·입찰 식별자, 중복, 허용 시간 |
| 외부 DSP → SSP | 요청·광고 영역 연결, 마감, 가격, 통화, 매체, 소재 |
| SSP → 자사 DSP | 송신자, 경매·입찰 연결, 사건 종류, 중복·역순·충돌 |
| 원본 시스템 → 자사 DSP | 데이터 출처, 버전, 완전성, 갱신 순서 |

송신자를 확인했다는 사실만으로 내용까지 올바른 것은 아니다. 신원 확인 뒤에도 도메인 규칙과 현재 상태를 다시 검증해야 한다.

## 확인된 위험

- 클라이언트 통지를 직접 과금하면 위조와 반복 호출로 광고주 예산을 소진시킬 수 있다.
- 공급자 통지의 낙찰가를 원본으로 사용하면 SSP가 결정한 가격과 다른 금액이 반영될 수 있다.
- 유효한 통지를 복사해 다시 보내는 재전송은 신원 확인만으로 막을 수 없다.
- 외부 DSP 응답이 다른 요청이나 광고 영역의 식별자를 사용하면 경매 결과가 오염될 수 있다.
- 이전 캠페인 데이터가 새 버전보다 늦게 도착하면 종료되거나 한도가 바뀐 캠페인이 다시 활성화될 수 있다.

## 동인 수정 사항

- 외부 입력이 낙찰과 금액 상태를 직접 확정하지 못하도록 권한과 검증 경계를 새 동인으로 추가해야 한다.
- 공급자 과금 통지의 가격은 SSP의 저장된 낙찰 결과와 대조해야 하며 불일치할 때의 처리를 정해야 한다.
- 신원 확인, 내용 무결성, 권한 확인, 재전송 방지는 서로 다른 검증으로 다뤄야 한다.
- 공개망 보안 운영이 범위 밖이어도 경계를 넘는 식별자와 금액 사건의 논리적 검증은 범위에 남아야 한다.
- 첫 성능 시험에서 TLS 비용을 제외하는 결정은 신뢰 경계 자체를 제거하지 않는다.

## ADR로 넘길 질문

- 각 서버 간 호출에서 송신자와 권한을 어떻게 확인할 것인가?
- 렌더링 통지가 실제 낙찰 광고에서 발생했음을 어떤 식별 정보로 증명할 것인가?
- 공급자 통지의 낙찰가와 SSP 결과가 다르면 어떤 사건으로 처리할 것인가?
- 재전송된 통지를 얼마 동안 어떤 식별자로 차단할 것인가?
- SSP에서 자사 DSP로 전달한 금액 사건의 위조·변조·중복을 어떻게 판정할 것인가?
- 캠페인 실행 데이터가 최신 원본 버전에서 왔음을 어떻게 확인할 것인가?
