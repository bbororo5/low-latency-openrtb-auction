# ADR-005 표준 과금 통지와 독립 장부

상태: 승인

근거: [아키텍처 중요 요구사항](../../requirements/quality.md), [ADR-001 분산 캠페인 예산 예약](ADR-001-distributed-budget-reservation.md)

## 1. 결정

SSP와 프로젝트 DSP는 금액 장부를 공유하지 않는다. 외부 계약은 OpenRTB의 `nurl`, `lurl`, `burl`만 사용하며 별도의 접수 증표·응답 객체·전용 헤더를 추가하지 않는다.

- DSP는 입찰 응답에 자기 통지 URL을 넣고 SSP는 URL을 해석하지 않고 호출한다.
- 연동 대상 SSP는 OpenRTB `burl`을 지원해야 한다. 지원 여부와 과금 발생 조건은 연동 전에 확인하며, 미지원 SSP를 사설 응답 규격으로 우회하지 않는다.
- DSP는 실제 노출까지 기다리는 표준 `Bid.exp`를 60초로 설정한다. 통지 전달·복구를 포함한 DSP 내부 예약 기한 95초와 혼동하지 않는다.
- `nurl`은 낙찰 사실만 알리며 금액을 바꾸지 않는다.
- `lurl`은 아직 유효한 잠재 지출을 해제한다. 통지가 없더라도 예약 만료가 같은 안전 결과를 만든다.
- `burl`은 SSP가 자기 정책에 따라 과금 가능하다고 판정한 시점의 통지다. DSP는 유효한 최초 통지만 확정 지출로 기록한다.
- DSP는 내부 사건을 PostgreSQL에 커밋한 뒤 본문 없는 HTTP `204`를 반환한다. 이것은 DSP의 내부 내구성 규칙이며 SSP가 해석해야 하는 회계 증표가 아니다.
- SSP의 재시도 정책은 OpenRTB가 보장하지 않는다. 재시도가 오면 DSP는 같은 URL을 멱등 처리하지만, 통지가 끝내 오지 않으면 예약을 만료하고 광고주에게 과금하지 않는다.
- `burl` 사건마다 지역 예산 원장을 갱신하지 않는다. 확정 지출과 반환액은 리스 단위로 집계해 멱등 정산한다.
- 양사의 청구·지출 자료는 일시적으로 다를 수 있다. 실제 청구서·송금과 업체 간 대사는 범위 밖이며, 통지 유실로 생긴 미과금은 초과·중복 과금보다 우선해 감수한다.

```mermaid

        RENDER["렌더링 판정"]
        SJ["SSP 지역 기록<br/>청구 근거·전달 결과"]
```

## 2. 사건과 책임

사건명은 회사 내부 사실만 표현한다. 상대 회사의 회계 확정을 뜻하지 않는다.

| 회사 | 내부 사건 | 의미 |
|---|---|---|
| SSP | `BillingClaimRecorded` | SSP가 렌더링을 자기 청구 근거로 인정함 |
| SSP | `BillingNoticeDelivered` | `burl` 호출이 일반 HTTP 성공으로 끝남 |
| SSP | `BillingNoticeUndelivered` | SSP 정책상 전달을 더 시도하지 않음 |
| DSP | `BillingNoticeRecorded` | 기한 안의 유효한 `burl`을 최초로 기록하고 광고주 지출 근거로 인정함 |
| DSP | `BillingNoticeExpired` | 처음 도착한 `burl`이 예약 기한을 지났으므로 금액에 반영하지 않음 |

DSP가 입찰 응답에 넣는 URL에는 외부가 해석하지 않는 불투명 토큰을 포함한다. 토큰은 예약·캠페인·리스·금액·만료 시각과 무결성 정보를 DSP 내부에서 검증할 수 있게 한다. 표준 치환 변수를 사용할 수 있지만 SSP 전용 확장 필드를 요구하지 않는다.

```text
https://dsp.example.com/openrtb/billing/{opaque-token}
```

동일 토큰의 중복 호출은 금액에 한 번만 영향을 준다. 정상 형식의 최초·중복·만료 통지는 처리 후 본문 없는 `204`를 반환한다. 위조·훼손되거나 해석할 수 없는 URL은 일반 HTTP 오류로 거부한다.

## 3. 리스 정산

지역 예산 원장은 리스를 발급할 때 전체 액면을 이미 가용액에서 제외한다. 따라서 `BillingNoticeRecorded`마다 다시 차감하면 같은 책임을 이중 반영하고 쓰기 경합만 늘어난다.

```text
리스 액면
= 확정 지출
+ 아직 유효한 잠재 지출
+ 증명된 미사용액
+ 장애로 불확실한 격리액
```

- 재리스 요청은 새 입찰 권한을 받는 절차일 뿐 기존 리스의 실제 과금을 증명하지 않는다.
- 리스 정산기는 지역 사건을 리스별로 집계한다.
- 모든 잠재 지출의 통지 가능 기한이 끝나면 확정 지출을 소비로, 나머지를 반환액으로 정한다.
- 지역 예산 원장은 `leaseId`와 정산 세대로 같은 정산을 한 번만 적용한다.
- 정산 전달이 실패하면 전체 리스가 계속 사용 불가능한 상태로 남고 재시도한다. 그동안 초과 지출은 발생하지 않는다.

## 4. 장애 계약

| 장애 시점 | 결과와 회복 |
|---|---|
| SSP가 `burl`을 호출하지 못함 | DSP는 예약 기한 뒤 잠재 지출을 해제하고 광고주에게 과금하지 않음 |
| DSP 기록 전 장애 | HTTP 성공을 반환하지 않음. SSP가 재시도하면 최초 처리하고, 재시도하지 않으면 만료 |
| DSP 기록 뒤 응답 유실 | 재호출되면 기존 사건을 확인해 금액을 다시 바꾸지 않고 `204` 반환 |
| DSP 인스턴스 장애 | 지역 장부에서 접수 사건을 복구하며, 리스는 안전한 정산 전까지 상위 가용액으로 돌아가지 않음 |
| 지역 장부 일시 중단 | 경매는 이미 발급한 로컬 권한으로 계속하고 통지 접수만 실패함 |
| 리스 정산기 장애 | 장부에서 리스별 집계를 다시 만들고 같은 정산 키로 재시도 |
| DSP 귀속 리전 재해 | 다른 리전이 같은 예약을 대신 확정하지 않음. 미확정액은 기한 뒤 보수적으로 정산 |

네트워크 실패만으로 상대 회사의 결과를 추정하지 않는다. 다만 외부 SSP가 표준 이상을 구현해야만 DSP 예산이 안전해지는 구조도 만들지 않는다.

## 5. 검토한 대안

| 대안 | 장점 | 탈락 이유 |
|---|---|---|
| 공유 장부·업체 간 분산 트랜잭션 | 하나의 상태처럼 보임 | 회사 책임과 장애를 결합하고 OpenRTB 연동 범위를 벗어남 |
| 전용 접수 증표와 재시도 계약 | 통지 결과를 사건별로 대조하기 쉬움 | SSP에 사설 응답 해석과 재시도를 요구해 호환성을 낮춤 |
| 표준 통지와 DSP 내부 멱등 처리 | 다른 업체에 추가 계약을 요구하지 않음 | 영구 통지 유실은 미과금 손실로 남음 |

표준 통지와 DSP 내부 멱등 처리를 선택한다. 업체 간 완전한 동시 회계보다 호환성, 중복 과금 방지와 광고주 보호를 우선한다.

## 6. 검증 조건

- 같은 `burl`을 동시에 여러 번 호출해도 `BillingNoticeRecorded`와 금액 효과는 한 건이다.
- 지역 장부 커밋 직후 애플리케이션을 종료해도 성공 응답한 사건을 복구할 수 있다.
- 응답을 유실시킨 뒤 같은 URL을 호출하면 금액을 다시 바꾸지 않고 일반 성공을 반환한다.
- `burl`을 보내지 않으면 예약 기한 뒤 잠재 지출이 반환되고 광고주 지출은 늘지 않는다.
- 만료 뒤 처음 도착한 `burl`은 기록만 남고 광고주 금액을 바꾸지 않는다.
- SSP는 응답 본문·전용 헤더·DSP 내부 사건명을 알지 않아도 연동할 수 있다.
- 사건 처리량과 무관하게 지역 예산 원장 쓰기는 리스 발급·반환·정산 횟수에 비례한다.
- 같은 리스 정산을 중복 전달해도 소비와 반환은 한 번만 반영된다.

## 7. 후속 작업

- ADR-006은 두 리전의 독립 실행과 지역 기록 배치를 반영한다.
- 지역 금액 사건 저장 기술과 배치는 [ADR-007](ADR-007-regional-monetary-event-store.md)을 따른다.
- 불투명 토큰 형식, HTTP 제한시간과 리스 정산 주기는 상세 설계에서 정한다.
