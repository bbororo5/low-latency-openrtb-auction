## 1. 핵심 동인

| ID | 동인 | 근거 |
|---|---|---|
| `동인-01` | 외부 호출, 캠페인 검색과 금액 예약을 포함한 경매를 p99 50ms 안에 끝내야 한다. | `ASR-01` |
| `동인-02` | 외부 실패와 초과 부하를 격리하고 최소 400 RPS의 지연과 정확성을 보호해야 한다. | `ASR-02` |
| `동인-03` | 분산된 DSP 인스턴스가 같은 캠페인 총예산을 초과하지 않고 잠재 지출을 95초 안에 수렴시켜야 한다. | `ASR-03` |
| `동인-04` | 분산 상태의 지연을 감수하면서도 캠페인 기간 전체의 예산 소비 속도를 제어해야 한다. | `ASR-04` |
| `동인-05` | 인스턴스·게이트웨이·AZ·리전 장애를 부분 영향으로 격리하고 성공으로 인정한 사실의 RPO 0을 지켜야 한다. | `ASR-05` |
| `동인-06` | 신뢰하지 않는 경계를 넘는 입찰·렌더링·금액 사건의 상대·연결·기한·중복을 검증해야 한다. | `ASR-06` |
| `동인-07` | 분산 흐름을 재현·설명하고 통신·저장·배포 방식을 도메인 의미와 독립적으로 바꿀 수 있어야 한다. | `ASR-07`, `ASR-08` |

## 2. 데이터와 불변식

| 데이터 | 소유자와 성격 | 보존 기준 |
|---|---|---|
| 경매 진행 상태 | SSP의 짧은 작업 상태 | 유실 시 요청은 실패할 수 있지만 180ms 뒤 결과를 만들지 않는다. |
| 공개한 낙찰 결과 | SSP가 렌더링과 통지에 사용하는 근거 | 60초 렌더링 판정과 통지 재시도에 필요한 동안 복구 가능해야 한다. |
| 렌더링 접수 | SSP가 검증할 외부 주장 | 최초 접수 시각과 낙찰 연결 근거를 복구해야 한다. |
| 통지 전달 상태 | SSP가 소유하는 비동기 작업 | DSP가 복구 가능하게 접수하거나 계약상 기한이 끝날 때까지 보존한다. |
| 캠페인 실행 데이터 | 프로젝트 DSP의 조회용 데이터 | 등록·중지·재개가 두 리전의 인스턴스에 일관되게 보인다. |
| 잠재·확정 지출 | 프로젝트 DSP의 금액 원본 | 캠페인 한도, 멱등 처리와 95초 만료를 복구할 수 있어야 한다. |
| 지표와 로그 | 검증용 파생 데이터 | 금액 원본을 대신하지 않고 흐름을 연결할 수 있어야 한다. |

```text
캠페인 확정 지출 + 유효한 잠재 지출 <= 캠페인 총예산
```

- 한 슬롯의 입찰은 하나의 잠재 지출을 만든다.
- 같은 캠페인이 여러 슬롯에 입찰하면 합계로 한도를 검사한다.
- 같은 논리 사건은 반복 전달되어도 금액 효과가 한 번만 발생한다.
- DSP 예약 전에 성공한 입찰은 없으며 예약 시각이 95초 기한의 기준이다.
- 처리 주체·인스턴스·배치가 바뀌어도 이미 만든 잠재 지출은 95초 수명주기를 그대로 따른다.
- 만료 전 `lurl`은 해제하고 `burl`은 확정하며, 사건이 없으면 만료 시 해제한다.
- 늦은 `burl`과 모순된 사건은 자동으로 금액을 다시 바꾸지 않는다.

## 3. 책임 경계

| 경계 | 책임 |
|---|---|
| SSP 경매 | 요청·DSP 응답 검증, 180ms 마감, 슬롯별 1가격 낙찰과 결과 보존 |
| SSP 렌더링·통지 | 렌더링 연결과 60초 판정, `nurl`·`lurl`·`burl`의 멱등 전달 |
| 외부 DSP 연동 | 외부 회사의 응답을 내부 경매 후보로 변환하고 실패를 격리 |
| 프로젝트 DSP 입찰 | 실행 데이터, 후보 검색, 페이싱, 캠페인·소재와 가격 선택 |
| 프로젝트 DSP 예산 | 잠재·확정 지출, 총예산 불변식과 95초 수명주기 |

SSP와 프로젝트 DSP는 서로 다른 회사의 독립 저장소를 사용한다. SSP는 경매와 과금 가능 렌더링을 확정하고 프로젝트 DSP의 공개 계약 주소로 `burl`을 보낸다. 프로젝트 DSP가 95초 안에 이를 복구 가능하게 접수할 때 광고주 캠페인 지출이 확정된다. 양쪽은 상대 내부 장부를 읽거나 호출하지 않고 같은 사건의 멱등 재시도로 응답 유실을 해소한다. 업체 간 지급·정산은 이 예산 원장과 분리한다.

## 4. 실패와 변화 원칙

- 외부 DSP 하나의 지연은 해당 입찰에서 끝난다.
- 기술 실패를 광고 없음이나 정상 `NO_BID`로 숨기지 않는다.
- 금액 경로가 불확실해도 예산 안전성을 증명할 수 있는 범위와 복구 가능한 과금 근거가 있으면 그 범위에서 프로젝트 DSP의 입찰을 계속한다. 둘 중 하나라도 없으면 프로젝트 DSP의 입찰을 중단하되 SSP는 외부 DSP와 경매를 계속한다.
- 인스턴스 수와 장애 영역이 바뀌어도 낙찰, 가격, 예약과 과금 의미는 바뀌지 않는다.
- JSON에서 이진 표현으로 바꾸어도 같은 합격 시험을 사용한다.
- 이전 버전과 새 버전이 함께 동작하는 동안 통신과 저장 상태가 호환되어야 한다.
- 정확히 한 번 전달을 가정하지 않고 여러 번 전달되어도 한 번만 반영한다.

## 5. ADR로 넘길 결정

1. 50ms를 SSP, 외부 호출, 프로젝트 DSP 게이트웨이·검색·예약에 어떻게 배분할 것인가?
2. 연결·대기열·처리 중 작업의 상한과 1,000 RPS 부하 차단을 어떻게 구현할 것인가?
3. 캠페인 실행 데이터를 어떤 구조와 버전으로 조회하고 중지·재개를 전파할 것인가?
4. 두 리전의 분산 인스턴스가 CPM을 정확하게 예약하면서 캠페인 총예산을 어떻게 보호할 것인가? [ADR-001](decisions/ADR-001-distributed-budget-reservation.md)은 계층형 에스크로 예산을 선택했다.
5. 선형 페이싱에서 확정·잠재 지출을 어떻게 반영하고 허용 편차를 어떻게 제어할 것인가?
6. `lurl`·`burl` 재시도, 멱등 접수, 95초 만료와 시계를 어떻게 보존할 것인가?
7. 전역·리전 원장의 분할·복제·쓰기 권한 차단은 [ADR-002](decisions/ADR-002-multi-region-ledger-topology.md)에서 검토한다. SSP·프로젝트 DSP·게이트웨이의 두 리전 트래픽 전환과 각자가 인정한 사실의 RPO 0은 후속 결정으로 남긴다.
8. 서버 간 상대 인증과 렌더링 통지의 낙찰 연결·변조·재사용 방지를 어떻게 구현할 것인가?
9. HTTP/JSON과 이진 표현, 순차 배포의 계약 호환성을 어떻게 유지할 것인가?
10. 성능·금액·페이싱·장애 합격 여부를 어떤 관측 정보로 자동 판정할 것인가?

ADR은 이 질문마다 대안, 선택 근거, 포기하는 성질과 검증 방법을 기록한다.
