# 운영과 실패 관점 검토

검토 대상: [아키텍처 동인](../architecture-drivers.md)

## 결론

완전 중단보다 느린 의존성, 자원 고갈과 재시도 증폭이 더 쉽게 전체 장애를 만든다. 실패 시 모든 요청을 성공시키는 것이 아니라 50ms와 금액 정합성을 보호하고 초과 작업을 명시적으로 포기해야 한다.

## 실패별 기대 동작

| 실패 | 허용되는 영향 | 허용되지 않는 영향 |
|---|---|---|
| 외부 DSP 지연·오류 | 해당 입찰 제외 | 다른 DSP와 경매 전체 실패 |
| SSP·DSP·게이트웨이 인스턴스 중단 | 처리 중 요청 일부 기술 실패 | 전체 중단, 인정된 사실 유실 |
| AZ 중단 | 일시적 성공률 저하 | 어느 1초 구간의 성공률 0%, RPO 위반 |
| 리전 중단 | 장애 리전 처리와 그 리전에 묶인 예산의 일시적 사용 중단 | 전역 경매 중단, 불확실한 예산의 즉시 재할당 |
| 연결·대기열 고갈 | 초과 요청의 빠른 기술 실패 | 무한 대기, 광고 없음으로 은폐 |
| 예산 예약 경로 불확실 | 안전하게 증명할 수 있는 한도만 사용하고 새 예약 중지 | 확인되지 않은 잔액 사용, 초과 지출 |
| DSP 과금 통지 접수 경로 불확실 | 복구 가능한 과금 근거와 적체 상한 안에서 예산 안전성을 증명할 수 있는 범위만 사용 | 근거 없는 새 예약, 안전 범위 밖 증가, 무기한 적체 |
| 통지 응답 유실 | 같은 사건 재시도 | 중복 금액 효과 |
| 95초까지 결과 없음 | 잠재 지출 해제, 늦은 과금 거부 | 무기한 예산 잠금 |

## 성능 저하 순서

1. 잘못된 낙찰과 금액 위반을 막는다.
2. 50ms 안에 끝낼 수 없는 작업을 더 받지 않는다.
3. 실패한 DSP·인스턴스·AZ·리전의 영향을 격리한다.
4. 1,000 RPS에서 최소 400 RPS를 보호하고 초과분을 거절한다.
5. 부하 감소 뒤 30초 안에 재시작 없이 회복한다.
6. 기존 잠재 지출은 통지 또는 95초 만료로 끝낸다.

## ADR로 넘길 질문

- 경매와 외부 호출에 시간 예산을 어떻게 배분할 것인가?
- 연결·대기열·처리 중 작업의 상한과 거절 기준은 무엇인가?
- DSP 과금 통지 경로의 안전성과 적체를 어떤 신호로 판단해 입찰을 중지·재개할 것인가?
- 인스턴스·AZ·리전 장애를 어떻게 감지하고 트래픽을 전환할 것인가?
- 복구 완료를 성공률, 지연, 적체와 금액 상태로 어떻게 판정할 것인가?
