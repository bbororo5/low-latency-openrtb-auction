# SSP 기술 기준선

상태: 확정 가능한 축만 반영

근거: [제품 요구사항](../../requirements/product.md), [아키텍처 중요 요구사항](../../requirements/quality.md), [SSP 컨테이너](../views/ssp-containers.md)

## 1. 지금 확정하는 기술

| 경계 | 기준선 | 근거 |
|---|---|---|
| 실행 환경 | Java 21 | 제품 제약이자 현재 코드 기준선이다. |
| 공급자·렌더링 API | HTTP/JSON | 초기 외부 표현이며 성능 비교의 기준선이다. |
| DSP 연동 | HTTP/JSON, OpenRTB 2.6 하위 규격 | 세 DSP가 같은 경매·통지 의미를 사용한다. |
| 경매 멱등 상태 | 제한된 프로세스 메모리 | 5초의 요청 키·지문·최초 결과만 보존해 Hot Path 외부 조회를 피한다. |
| 렌더링 증표 | 버전형 이진 페이로드 + AEAD + Base64URL | 서버 낙찰 상태와 외부 키 조회 없이 2초 기한의 낙찰·과금 정보를 어느 리전에서나 검증한다. |
| 리전별 SSP 청구 기록 DB | PostgreSQL | ADR-007에서 성공 응답 전 내구 기록과 멱등 판정의 저장 기술로 승인했다. |
| 배포 전제 | 두 능동 리전, 리전별 다중 AZ | ADR-006의 장애 격리 경계다. |

이 기준은 SSP의 외부 계약과 데이터 권위를 고정한다. HTTP 서버·클라이언트 라이브러리와 동시성 모델은 고정하지 않는다.

## 2. 컴포넌트 뒤에 정할 기술

| 선택 | 먼저 확인할 컴포넌트 압력 |
|---|---|
| HTTP 서버 실행 모델 | 우선 공유 자원으로 측정하고 경매·렌더링 경합이 확인될 때 어떤 격리 단위가 필요한가 |
| DSP HTTP 클라이언트 | DSP별 연결·동시성·기한·취소를 독립적으로 제어할 수 있는가 |
| 진입·부하 분산 제품 | 다중 AZ 건강 확인과 과부하 차단 책임을 어디까지 맡길 것인가 |
| 확장 시 요청 키 라우팅 | 같은 요청 키의 실행 소유자를 외부 상태 조회 없이 어떻게 하나로 유지할 것인가 |
| PostgreSQL 접근 방식 | 렌더링 경로의 내구 기록이 경매 실행 자원을 막지 않는가 |
| 렌더링 증표 구현 | 확정한 AEAD 계약을 낮은 할당과 예측 가능한 크기로 구현하는 코덱·암호 라이브러리는 무엇인가 |
| 관측 라이브러리 | 경매·DSP 호출·렌더링·저장·통지를 같은 식별자로 연결할 수 있는가 |

## 3. 선택 순서

1. SSP 협력 메시지를 책임으로 묶고 C3 컴포넌트를 정한다.
2. 컴포넌트별 동시성·저장·장애 경계를 정한다.
3. 실행 모델을 작은 기술 검증으로 비교한다.
4. 합격한 모델에 맞춰 라이브러리와 제품을 선택한다.

라이브러리 선택이 C2 경계나 데이터 권위를 바꾸면 먼저 아키텍처 문서를 재검토한다.
