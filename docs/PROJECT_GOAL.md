# Project Goal

## 1. 목표

이 프로젝트의 목표는 OpenRTB 2.6의 입찰 요청/응답 흐름을 기반으로, **제한 시간 안에서 여러 Bidder의 응답을 수집하고 낙찰자를 결정하는 백엔드 시스템**을 단계적으로 구현하는 것이다.

이 프로젝트는 광고 도메인 전체를 구현하기 위한 것이 아니다. 핵심은 RTB라는 문제 상황을 이용해 다음 역량을 드러내는 것이다.

- 명세 기반 HTTP API 이해
- read-heavy 처리 흐름 이해
- 고동시성 요청 처리
- timeout과 실패 케이스 처리
- 테스트와 성능 측정
- 측정 결과에 기반한 최적화
- 문제 해결 과정 문서화

## 2. 핵심 문제

하나의 `BidRequest`가 들어왔을 때 여러 Bidder가 동시에 응답할 수 있다. 하지만 모든 Bidder가 빠르고 정상적으로 응답한다는 보장은 없다.

이 프로젝트에서 다루려는 핵심 문제는 다음과 같다.

> 제한 시간 안에서 여러 Bidder의 응답을 수집하고, no-bid, timeout, invalid bid를 제외한 뒤 유효한 입찰 중 낙찰자를 결정하는 것.

## 3. 현재 범위

현재 목표 설정 단계에서는 아래 범위까지만 확정한다.

- OpenRTB 2.6의 `BidRequest` / `BidResponse` 흐름을 기반으로 한다.
- 시스템의 관심사는 낙찰자 결정까지이다.
- 실제 광고 노출, 클릭, 과금, 리포팅은 제외한다.
- 상세 컴포넌트 구조와 기술 선택은 PRD와 Tech Spec 단계에서 결정한다.
- 최적화 방식은 초기 구현과 측정 이후 결정한다.

## 4. 보여주려는 역량

### 기본기

- HTTP 요청/응답 처리
- JSON 기반 API 계약
- 입력 검증
- 예외 처리
- 테스트 작성
- 문서화

### 강화된 기본기

- timeout 처리
- 여러 외부 요청의 동시 처리
- 실패 케이스 분류
- read-heavy 처리 경로 설계
- 성능 측정 지표 정의

### 심화 역량

- p95/p99 latency 분석
- 병목 지점 탐색
- 측정 기반 최적화
- 설계 의사결정 기록
- trade-off 설명

## 5. 진행 방식

프로젝트는 아래 순서로 진행한다.

1. Project Goal
2. PRD
3. Tech Spec
4. ADR
5. Feature Development
6. Test & Validation
7. Performance Measurement
8. Optimization
9. Retrospective

각 단계에서는 현재 단계에 필요한 결정만 내린다. 이후 단계의 결정을 앞당겨 확정하지 않는다.
